# external_trust_signals.id — IDENTITY Fix (2026-04-21)

## Problem

`zarq.external_trust_signals.id` was declared `integer NOT NULL` with no
`DEFAULT`. Every `INSERT` that omitted an explicit `id` hit a NOT NULL
violation. `community-signals` crawler (LaunchAgent `com.nerq.community-signals`,
Sundays 08:00 UTC) writes via `dual_write.dual_execute`, which translates
SQLite `INSERT OR REPLACE` into PostgreSQL `INSERT ... ON CONFLICT` — but
neither path ever supplied `id`. Result: every row's PG mirror failed with

```
null value in column "id" of relation "external_trust_signals"
violates not-null constraint
```

The crawler also eventually hit `sqlite3.OperationalError: database is
locked` and aborted mid-run. The `id` NOT NULL issue is the primary
unblocker.

## Fix

One DDL statement on Nbg primary (replicates to Hel + Mac):

```sql
ALTER TABLE zarq.external_trust_signals
  ALTER COLUMN id ADD GENERATED BY DEFAULT AS IDENTITY
  (START WITH 51828);
```

`51828 = MAX(id) + 1` at the time of the change.

`GENERATED BY DEFAULT` (not `ALWAYS`) is important: existing dual-write
code that supplies explicit `id` values (e.g. from the SQLite autoincrement
source of truth) still works; new inserts that omit `id` get one auto-assigned.

## Verification

```sql
SELECT is_identity, identity_generation, identity_start
FROM information_schema.columns
WHERE table_schema='zarq' AND table_name='external_trust_signals'
  AND column_name='id';
-- is_identity=YES, identity_generation=BY DEFAULT, identity_start=51828

INSERT INTO zarq.external_trust_signals (agent_name, source, signal_name, signal_value)
VALUES ('test_fix', 'manual', 'test', 1.0) RETURNING id;
-- returned 51828 (cleaned up)
```

Regression suite: 12/12 PASS (2026-04-21 11:23 CEST).
