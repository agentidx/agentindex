## Evidence ranking (most → least decisive)

### 1. Slow-query smoking gun (Nbg `pg_stat_activity`)

Captured between intermittent Nbg connect failures:

```
   pid   | state  |     runtime     |                      q
---------+--------+-----------------+-----------------------------------
 3108767 | active | 00:01:05.056845 | SELECT id FROM software_registry  +
                                       WHERE lower(name) LIKE
 3108602 | active | 00:00:46.030674 | SELECT id FROM software_registry  +
                                       WHERE lower(name) LIKE
 3108758 | active | 00:00:44.547992 | SELECT id FROM software_registry  +
                                       WHERE lower(name) LIKE
 3108606 | active | 00:00:41.358200 | SELECT id FROM software_registry  +
                                       WHERE lower(name) LIKE
 3108766 | active | 00:00:01.330593 | SELECT id FROM software_registry  +
                                       WHERE lower(name) LIKE
```

**5 concurrent identical queries**, oldest at **65 seconds** runtime.
This is a textbook PG saturation pattern. PgBouncer's
`query_timeout = 60` will kill these (manifest as "server conn
crashed?"); `query_wait_timeout = 30` will reject every client whose
query is waiting in the pool.

**Origin of the query:**

```
agentindex/preflight.py:250:
    SELECT ... FROM software_registry
    WHERE registry = :reg AND lower(name) LIKE lower(:pat)
```

`software_registry` is the Nerq 7.5M+ entity corpus. `registry` is
likely indexed; `lower(name) LIKE` is **not** — no functional index
on `lower(name)`, no `pattern_ops` opclass. Result: per-call full table
scan filtered to the registry partition. With concurrent calls (the
universal `preflight_check` MCP tool, plus `/preflight/*` HTTP routes),
this saturates fast.

### 5. Infrastructure_alerts — META-BUG

```sql
SELECT * FROM zarq.infrastructure_alerts WHERE resolved_at IS NULL;
-- (0 rows)
```

**The healthcheck we built this morning doesn't catch this failure
mode.** Our probe is `socket.create_connection((host, port))` — pure
TCP. Nbg accepts TCP connections fine; what fails is *per-query
execution* (slow / dropped mid-flight). The healthcheck reports
"all-clear" because the port is reachable.

This is a meta-bug. R7 will continue to escalate silently as far as
`zarq.infrastructure_alerts` is concerned. The healthcheck should also
do a `SELECT 1` with a short statement timeout, not just a TCP
handshake.

### 6. Direct evidence from `api_error.log`

```
Failed to create user_reviews table:
  (psycopg2.OperationalError) server closed the connection unexpectedly
  This probably means the server terminated abnormally
  before or while processing the request.
```

This is the API worker boot path. Every worker tries to `CREATE TABLE
IF NOT EXISTS user_reviews` at startup. With Nbg's PG dropping
connections mid-statement (the `lower(name) LIKE` queries are
saturating it), the boot DDL fails. The worker dies. Uvicorn restarts.
Loop.

## Verdict on the three diagnostic questions

| Question | Answer |
|---|---|
| Stable degraded or progressively escalating? | **Progressively escalating.** +8 restarts in 35 min; localhost:8000 now timing out; direct Nbg PG connects also timing out. |
| Immediate cause? | **Slow `lower(name) LIKE` on software_registry**, hitting PgBouncer's `query_timeout=60s` and rejecting waiters at `query_wait_timeout=30s`. Multiple concurrent calls amplify the saturation. |
| Is `zarq.infrastructure_alerts` blind? | **Yes.** Our TCP-only probe can't detect per-query failures. Meta-bug. |
| Safe to expose api.zarq.ai now? | **No.** Adding ~60 endpoints to a backend that's restart-looping accelerates the failure. R1 must wait. |

## Recommended next actions (separate workstream, not this prompt)

1. **Immediate (≤5 min)**: Stop the worker-boot DDL retry storm. The
   `CREATE TABLE IF NOT EXISTS user_reviews` at worker boot is amplifying
   the saturation. Move it to a one-time idempotent migration at deploy
   time rather than per-worker-boot. Smallest-blast-radius change.

2. **Root cause (≤30 min)**: Add a functional index on
   `lower(software_registry.name)` (probably with `text_pattern_ops` for
   `LIKE` queries):

   ```sql
   CREATE INDEX CONCURRENTLY idx_software_registry_lower_name
       ON software_registry (registry, lower(name) text_pattern_ops);
   ```

   `CONCURRENTLY` so it doesn't block writes during creation. This is
   directly the R7 fix and is *not* one of the ADR-003a HA prerequisites
   — it's a single-table indexing issue.

3. **Meta-bug fix** (any time): extend `scripts/infra_healthcheck.py` to
   also run `SELECT 1` with a 4-second statement_timeout, not just TCP.

4. **Then** R1 (api.zarq.ai ingress). At that point the backend can
   handle the added traffic.
